# v0.2.6 Critical Fixes — Design Spec

**Date:** 2026-04-16
**Author:** Koti (with Claude)
**Status:** Draft, pending review
**Release target:** v0.2.6

Two independent critical fixes that must land together before the desktop release. Bundled into one spec because the release pipeline is a single commit → tag → build cycle; splitting would produce two half-releases.

## Problem Statement

### Issue 1 — Dev context menu leaking into release builds (preventative hardening)

`Cargo.toml` attempts to gate the Tauri `devtools` Cargo feature on `debug_assertions` via `[target.'cfg(debug_assertions)'.dependencies]`. Cargo does not actually support `debug_assertions` as a target cfg predicate — the block is a silent no-op, emits a warning on every build (`warning: Found 'debug_assertions' in target.'cfg(...)'.dependencies. This value is not supported…`), and gives a false sense of protection. Today's release binaries are actually safe because Tauri 2's WKWebView inspector gate is internal (keyed on Rust's built-in `debug_assertions`), not our Cargo feature. But if anyone on the team ever adds `"devtools"` to the base `features` array in a moment of debugging, there is nothing stopping them from silently shipping the inspector to end users.

### Issue 2 — Silent cluster identity substitution (data-integrity P0)

When a user clicks "Connect" on a cluster whose stored kubeconfig file no longer exists, the backend silently falls back to the user's system `~/.kube/config` and builds a client from whatever context it finds. The cluster DB row still says "AWS" (Name + Context from when the user added it), so the frontend labels the dashboard as AWS while the actual data is whatever cluster the system kubeconfig currently points to — typically `docker-desktop`.

**Root cause — backend:** `ReconnectCluster` at `cluster_service.go:812-823`:

```go
path := c.KubeconfigPath
if path != "" {
    if _, err := os.Stat(path); err != nil {
        path = ""
    }
}
if path == "" {
    home, _ := os.UserHomeDir()
    if home != "" {
        path = filepath.Join(home, ".kube", "config")  // ← silent identity swap
    }
}
client, err := k8s.NewClient(path, c.Context)
```

**Secondary root cause — orphan accumulation:** `kubeconfigwatch.partition()` at `sync.go:144-155` filters `c.Source != "kubeconfig"` and skips upload-sourced clusters from orphan cleanup entirely. Upload-sourced rows are permanently sticky, which guarantees a long tail of stale clusters that feed into the Issue 2 bug path.

**Tertiary root cause — frontend defense:** `ClusterConnect.handleConnect` navigates to `/dashboard` optimistically after calling setActiveCluster, without awaiting a confirmation that the backend actually connected successfully. If the backend silently substitutes a cluster (Issue 2 backend bug) or returns an error (post-fix), the user lands on Dashboard either way with no warning.

User has reported this class of bug multiple times. Prior fixes only addressed the sidebar counters symptom; the underlying identity violation persisted. This spec addresses the root causes.

## Goal

Make it impossible for Kubilitics to display data from one cluster under another cluster's identity. "Missing kubeconfig" means "missing cluster" — return an error, show the error, let the user decide to reconnect or remove. Never silently swap.

## Non-Goals

- Adding an "Unregister cluster" button on the Fleet page. The existing DELETE endpoint is out of scope for this release.
- Restructuring the cluster registration flow or consolidating `AddCluster` / `AddClusterFromBytes`.
- Changing the backend's cluster DB schema.
- UI work beyond the toast + error banner on `handleConnect` failure.

### Backend fix 2a — Kill the silent kubeconfig fallback

Current code at `cluster_service.go:811-823` silently falls through to `~/.kube/config` when the cluster's stored kubeconfig is missing. Replace with strict validation:

```go
path := c.KubeconfigPath
if path == "" {
    c.Status = "disconnected"
    _ = s.repo.Update(ctx, c)
    return c, fmt.Errorf("cluster %s has no stored kubeconfig path — reconnect or remove it", id)
}
if info, err := os.Stat(path); err != nil {
    if errors.Is(err, fs.ErrNotExist) {
        c.Status = "disconnected"
        _ = s.repo.Update(ctx, c)
        return c, fmt.Errorf("cluster %s kubeconfig file no longer exists at %s — reconnect or remove it", id, path)
    }
    // Transient I/O error (permission, network-mount unreachable) — report but do not mark disconnected.
    return c, fmt.Errorf("cluster %s kubeconfig file unreadable at %s: %w", id, path, err)
} else if info.IsDir() {
    c.Status = "disconnected"
    _ = s.repo.Update(ctx, c)
    return c, fmt.Errorf("cluster %s kubeconfig path is a directory, not a file: %s", id, path)
}
```

Everything from line 830 (`client, err := k8s.NewClient(path, c.Context)`) onward stays unchanged.

Add `"errors"` and `"io/fs"` to the imports if not already present.

### Backend fix 2b — Extend orphan detection to upload-sourced clusters

Current `partition` at `sync.go:144-155` filters out non-kubeconfig rows entirely. Replace with:

```go
func partition(persisted []*models.Cluster, contextSet map[string]struct{}) (eligible []*models.Cluster, orphans []*models.Cluster) {
    for _, c := range persisted {
        if c.Source == "kubeconfig" {
            eligible = append(eligible, c)
            if _, ok := contextSet[c.Context]; !ok {
                orphans = append(orphans, c)
            }
            continue
        }
        // Non-kubeconfig sources (upload, etc.): orphan when the stored file is missing.
        if c.KubeconfigPath == "" {
            eligible = append(eligible, c)
            orphans = append(orphans, c)
            continue
        }
        if _, err := os.Stat(c.KubeconfigPath); err != nil {
            if errors.Is(err, fs.ErrNotExist) {
                eligible = append(eligible, c)
                orphans = append(orphans, c)
            }
            // Transient I/O error → leave alone, next sync will re-check.
        }
    }
    return eligible, orphans
}
```

Add `"errors"`, `"io/fs"`, and `"os"` to the imports if not already present. Safety cap behavior is unchanged — it still gates mass deletion.

### Frontend fix 2c — Defense in depth in `handleConnect`

Current `handleConnect` at `ClusterConnect.tsx:436-473` calls `setActiveCluster` and navigates immediately. Change to: `await reconnectCluster(baseUrl, cluster.id)` (already exported from `src/services/api/clusters.ts:128`, which POSTs to `/api/v1/clusters/{clusterId}/reconnect` and maps onto the backend `ReconnectCluster` we just hardened). On success → continue to navigate. On error → catch, show a destructive `toast.error()` with the backend's error message (e.g. *"Cluster `<name>` is unreachable: <error>. Reconnect or remove it from Fleet."*), mark the card as errored via local state, and do NOT navigate.

This reuses existing infrastructure: the reconnect API already exists, the toast helper already exists, `ClusterConnect` already has card state. ~15 lines added. No new routes, no new endpoints.

### Tests

**`cluster_service_test.go` — new cases:**
- `ReconnectCluster` with cluster row `KubeconfigPath=""` → returns error containing "no stored kubeconfig path", does not populate `s.clients`, marks cluster disconnected in DB
- `ReconnectCluster` with `KubeconfigPath` pointing at a temp file that was deleted → returns error containing "no longer exists", cluster marked disconnected
- `ReconnectCluster` with `KubeconfigPath` pointing at a directory → returns error containing "is a directory"
- `ReconnectCluster` with a valid temp kubeconfig → still succeeds (regression safeguard)
- `ReconnectCluster` with a transient permission error → returns error but does NOT mark disconnected

**`sync_test.go` — new cases for `partition`:**
- kubeconfig-sourced, context missing from watched kubeconfigs → orphan (existing behavior regression-tested)
- upload-sourced, `KubeconfigPath=""` → orphan (new)
- upload-sourced, file present on disk → NOT orphan (new)
- upload-sourced, file missing from disk → orphan (new)
- upload-sourced, permission-denied I/O error (simulated) → NOT orphan (new — transient)
- Mixed list of all above → only the correct subset is returned

**Frontend test for `handleConnect` failure path:** mock `useClustersFromBackend` to return a cluster with `status: 'disconnected'` after the click; assert `navigate` was not called and a toast was fired.

### Rebuild + release sequence

1. Land all three cluster-identity backend/frontend fixes AND the Tauri config hardening on main in a single commit.
2. Run backend `go test ./...` — confirm all new cases pass.
3. Run frontend `npx vitest run` — confirm no regressions.
4. Rebuild the Go sidecar binary (`make backend` or equivalent) so the desktop release bundles the fixed binary.
5. Version files are already bumped to 0.2.6 — commit is staged.
6. Commit, push, tag `v0.2.6`, push tag. GitHub Actions `release.yml` picks it up and produces the desktop artifacts.
7. User downloads v0.2.6 and validates: (a) right-click shows no Inspect Element, (b) clicking Connect on a cluster whose kubeconfig was removed externally shows an error and stays on the connect page instead of silently loading Dashboard.

## Risks

- **Breaking existing users whose stored kubeconfig lives on a network mount.** Mitigated by only treating `fs.ErrNotExist` as disconnected — transient I/O errors are reported but not persisted.
- **Orphan cleanup suddenly nuking a legitimate upload-sourced cluster whose file was moved, not deleted.** The safety cap (`KubeconfigSyncMaxAbsoluteRemovals`, default 10; `KubeconfigSyncMaxRemovalRatio`, default 0.5) still applies — a mass wipe would trip the cap and require operator review.
- **Frontend `handleConnect` regression in the happy path.** Mitigated by keeping the existing success branch unchanged — the added code only runs when the backend reports failure.

## Out of Scope

- Adding a Fleet-page "Unregister cluster" action (tracked separately).
- Surfacing stale-cluster warnings proactively in the cluster picker before the user clicks.
- Migrating cluster storage from per-file kubeconfigs to a central database-blob scheme.
- Fixing sidebar counters again — already covered by commit `7ff5d6e`.
